# <シナリオのタイトル>

**対象読者:** admin | user
**動作前提:** 出荷済み・全緑の成果物のみ（オフライン優先）。それに当てはまらない
手順は `coverage.yaml` で gap として明示し、決して偽装しない。

## このシナリオで体験できること

1段落で: 読者が実際に手を動かして得られる具体的な成果と、3軸（コスト / 性能 / 精度）の
どれに触れるか。どの手順が今日動き、どの手順が意図的に gap に突き当たるかを冒頭で明示する。
gap に突き当たること自体が学びの一部である。

## 前提

- 環境（オフラインかライブか、どの出荷済みコマンド/テストを使うか）。
- **責務境界（はっきり書く）:** 測定・評価・可用性目標・backend 適合は *利用者* の責務である。
  このシナリオが提供するのは *機構*（決定的なスクリプト、指標の定義、純粋な採点 fold）であって、
  あなたのワークロードに対する監査済みの結果ではない。

## 手順

番号付き・コピーして実行できる形。各手順は `coverage.yaml` の1エントリに対応する。今すぐ実行できる
手順にはコマンドと期待される出力の形を示す。gap に突き当たる手順はそれを明示し、tracking issue を
指し示す。「X を測定してみる → 今日 gateway が出せるのは Y だけ → その gap が issue #NNN」。

## 測定

何を、どの機構で測定するか。ある指標がまだ gateway から emit されていない場合、この節は
「その指標はここで *テンプレートとして定義* されているだけで、emit は gap である」と明記する。
数字を捏造しない。

## 期待結果

決定的な結果（数字が散文に現れる箇所は `run.py` / CI テストで pin し、docs がサイレントに
ズレないようにする。Savings デモと同じ姿勢）。

## Coverage と gap

`coverage.yaml` が符号化している内容の散文サマリ: どの手順が `covered` か、どれが
`not-implemented`（issue リンク付き）か、どれが `user-responsibility` か。
全シナリオを横断した集計ビューは自動生成の `scenarios/COVERAGE.md` にある。
